# Ozubulu Digital Platform

# Design Language & Design System

## Part II — Components

**Version:** 1.0

**Status:** Draft

---

# Purpose

This document defines the reusable interface components used throughout the Ozubulu Digital Platform.

Every component should be:

* Accessible
* Responsive
* Consistent
* Reusable
* Easy to maintain

Components should emphasise content rather than compete with it.

---

# Header & Navigation

The navigation is the visitor's primary way of exploring the platform.

It should remain simple, predictable, and uncluttered.

## Structure

The main navigation contains **five primary links**.

1. Discover
2. Heritage
3. Community
4. Explore
5. Support

The Ozubulu wordmark appears on the left.

Search and a mobile menu button appear on the right.

---

## Mega Menus

Mega menus provide access to major sections without overwhelming visitors.

### Discover

* About Ozubulu
* History
* Communities
* Villages
* Map (Future)

### Heritage

* Culture
* Traditions
* Festivals (Future)
* Historical Articles
* Gallery

### Community

* Notable People
* Institutions (Future)
* Development Projects (Future)
* News

### Explore

* Search
* Timeline (Future)
* Archives (Future)
* Sources

### Support

* Contribute
* Submit a Correction
* Contact
* Support the Project

Mega menus should open on hover (desktop) and tap (mobile).

Animations should be subtle and quick.

# Hero Section

The homepage hero is the emotional introduction to the platform.

It should immediately communicate place, identity, and purpose.

## Structure

Background:

A full-screen cinematic video showing Ozubulu.

Suggested scenes include:

* Sunrise over the community
* Roads and village entrances
* Palm trees
* Community life
* Traditional landmarks
* Daily activities
* Drone footage
* Cultural moments

The video should be muted and loop seamlessly.

A dark overlay improves text readability.

---

## Hero Content

### Primary Heading

**Welcome to Ozubulu**

### Tagline

*"Where history lives, culture thrives, and a community's story is preserved for generations."*

### Supporting Text

A brief introduction explaining that the platform is the trusted digital home of Ozubulu's history, people, and heritage.

Keep this under three sentences.

Every major section follows the same pattern.

* Eyebrow label (optional)
* Heading
* Short introduction
* Content

# Buttons

Buttons should feel confident without demanding attention.

## Primary

Background:

Camwood

Text:

White

Used for:

* Main calls to action
* Submit actions
* Navigation

---

## Secondary

Transparent background.

Camwood border.

Used for secondary actions.

---

## Tertiary

Text only.

Used for inline actions and supporting links.

---

# Cards

Cards organise information into manageable sections.

Cards should have:

* Linen White background
* Soft Clay border
* Large border radius
* Gentle shadow
* Comfortable spacing

Cards should never feel heavy.

---

## Village Card

Contains:

* Featured image
* Village name
* Community
* Short description
* "Learn More" action

---

## Person Card

Contains:

* Portrait
* Name
* Occupation
* Community
* Verification badge (when applicable)

---

## Historical Article Card

Contains:

* Cover image
* Title
* Summary
* Publication date
* Reading time

---

# Gallery

Photography plays a central role in preserving Ozubulu's story.

Gallery images should:

* Maintain original aspect ratios
* Include captions
* Include photographer attribution
* Include alt text

Lightbox viewing should support keyboard navigation.

---

# Timeline

Historical timelines should use:

* Bronze Heritage accents
* Vertical line
* Date markers
* Expandable events
* Source references

Timelines should encourage exploration without overwhelming the visitor.

---

# Verification Badge

Verification status should always be visible.

Examples:

Verified

Multiple Sources

Single Source

Oral Tradition

Under Review

Disputed

Badges should use colour alongside text.

Colour alone should never communicate meaning.

---

# Search

Search should be available from every page.

Search results should clearly identify the content type:

* Community
* Village
* Person
* Article

Filters should remain simple.

---

# Forms

Forms should prioritise clarity.

Every field should include:

* Label
* Help text where needed
* Validation feedback
* Accessible focus states

Long forms should be broken into logical sections.

---

# Tables

Tables should be used only when they improve understanding.

Examples:

* Historical records
* Population data
* Reference lists

Tables must remain responsive.

---

# Empty States

Every empty state should explain:

* Why nothing is shown.
* What the visitor can do next.

Avoid dead ends.

---

# Error Pages

404

Explain that the requested page could not be found.

Offer links back to:

* Home
* Search
* Communities

500

Apologise briefly.

Encourage the visitor to try again later.

---

# Footer

The footer represents permanence.

Sections include:

* About
* Explore
* Community
* Support
* Contact

Secondary area:

* Copyright
* Editorial Policy
* Privacy
* GitHub
* Version number

---

# Accessibility

Every component must support:

* Keyboard navigation
* Screen readers
* Visible focus indicators
* Colour contrast
* Reduced motion preferences

Accessibility is part of the design—not an enhancement.

---

# Component Philosophy

Every component should answer one question:

**Does this help visitors discover, understand, and preserve the story of Ozubulu?**

If the answer is no, the component should be simplified or removed.
